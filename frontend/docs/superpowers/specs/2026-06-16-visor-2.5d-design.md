# Diseño: Visor 2.5D tipo Blender para ScRmhoot

## Fecha: 2026-06-16

## 1. Motivación

Reemplazar el visor 2D actual (Canvas + atlas procedural) por un visor 3D con 3 viewports ortográficos tipo Blender (Axial, Coronal, Sagital). El usuario puede cargar imágenes PNG/JPG propias, aplicar filtros MRI simulados (T1, T2, FLAIR, PD, DWI), y manipular el FOV como un cubo 3D que se refleja en todos los viewports.

## 2. Stack

- **Three.js** (renderizado 3D, 3 cámaras ortográficas)
- **Canvas 2D offscreen** (filtros MRI sobre texturas)
- **React 19** (sin migrar a Preact — no vale la pena para proyecto inconcluso)
- **TailwindCSS 4** (UI)
- **Lógica pura** en `src/core/` (separada de React/Three.js)

## 4. FOV Math (corregido)

```typescript
interface FOVBox {
  position: [number, number, number];  // centro 3D
  size: {
    read: number;      // FOV Read (eje X)
    phase: number;     // FOV Phase (eje Y)
    depth: number;      // Slice Coverage (eje Z)
  };
}

// Fórmula correcta:
function calculateSliceCoverage(thickness: number, slices: number, gap: number): number {
  return (thickness * slices) + (gap * (slices - 1));
}

// depth = (Slice Thickness × Slices) + (Gap × (Slices - 1))
// NO depth = FOV Read × FOV Phase × Slice Coverage
```

## 5. Estado: React vs Imperativo (anti-lag)

Durante drag, las transformaciones aplican directamente a `mesh.position` de Three.js (imperativo). React state solo actualiza en `pointerup` (valor final) o throttled (100ms) para los inputs numéricos del form.

```typescript
interface FOVState {
  committed: { position: Vector3, size: Vector3 };  // React state
  dragging: { position: Vector3, size: Vector3 };   // useRef (60fps)
}
```

## 7. Sistema de coordenadas (RAS, future-proof)

RAS por defecto:
- `+X` = Right
- `+Y` = Anterior
- `+Z` = Superior

Las orientaciones de los planos se definen con matrices 4×4 (vectores directores), no ejes fijos:

```typescript
interface SliceOrientation {
  rowVector: [number, number, number];     // DICOM ImageOrientationPatient[0..2]
  colVector: [number, number, number];     // DICOM ImageOrientationPatient[3..5]
  normalVector: [number, number, number];  // cross(row, col)
  position: [number, number, number];      // DICOM ImagePositionPatient
}

// Axial: row=[1,0,0], col=[0,1,0], normal=[0,0,1]
// Coronal: row=[1,0,0], col=[0,0,1], normal=[0,1,0]
// Sagittal: row=[0,1,0], col=[0,0,1], normal=[1,0,0]
// Oblicuos (futuro): row=[0.5,0.5,0.707], col=[-0.5,0.5,0.707], ...
```

## 8. Layout definitivo

**2×2 Quad View (tipo Blender)**

```
┌──────────────┬──────────────┐
│   Axial      │              │
│  (top-down)  │              │
│              │   Sagittal   │
├──────────────┤   (right)    │
│   Coronal    │              │
│  (front)     │              │
└──────────────┴──────────────┘
```

- Axial y Coronal: 50% altura cada uno
- Sagittal: 50% ancho, 100% altura

## 9. Carga de imágenes

- Botón "Cargar Imagen" por plano (3 botones, uno por viewport)
- File picker: `image/png, image/jpeg`
- Crear `THREE.Texture` desde `ImageBitmap` o `Canvas` offscreen
- Asignar al mesh del plano correspondiente
- Si hay filtro activo: aplicar filtro antes de texturizar

## 10. Filtros MRI

Lógica actual (`applyT1Filter`, `applyT2Filter`, etc.) adaptada para operar sobre `ImageData` de la imagen subida (o atlas):

1. Tomar imagen fuente (canvas offscreen)
2. Aplicar filtro (manipulación de píxeles)
3. Crear nueva `THREE.Texture` desde el canvas resultante
4. Asignar al mesh del plano

Los 3 viewports se actualizan automáticamente al cambiar la textura.

## 11. Window / Level (W/L)

Parámetro independiente de los filtros:
- **Window Center (WC)** = brillo (valor central del rango)
- **Window Width (WW)** = contraste (rango de valores)

UI: slider en `WindowLevelSlider.tsx` (pestaña "System" o nuevo panel).

```typescript
// core/windowing.ts
function applyWindowLevel(pixelValue: number, wc: number, ww: number): number {
  const min = wc - ww / 2;
  const max = wc + ww / 2;
  return clamp((pixelValue - min) / (max - min) * 255, 0, 255);
}
```

## 12. Slice ↔ FOV sync

El FOV cubo define la posición de las slices:

```typescript
// Dentro del FOV, slices distribuidas en Z (normal del plano):
function calculateSlicePositions(fov: FOVBox, sliceCount: number, thickness: number, gap: number): Vector3[] {
  const start = fov.position.z - fov.size.depth / 2;
  const positions = [];
  for (let i = 0; i < sliceCount; i++) {
    const offset = (thickness + gap) * i;
    positions.push(new Vector3(fov.position.x, fov.position.y, start + thickness / 2 + offset));
  }
  return positions;
}

// Visualización:
// - En la vista 3D (viewport): todas las slices como líneas paralelas
// - En el plano activo: solo la slice central del FOV
```

## 14. Notas

- **Preact**: No se migra. El esfuerzo no compensa para un proyecto inconcluso.
- **DICOM**: No se implementa en esta fase. El sistema de coordenadas con matrices 4×4 está preparado para futuro.
- **Atlas procedural**: Se mantiene como textura por defecto. Se genera el atlas 2D actual y se usa como textura de Three.js.
- **Saturation bands**: Se renderizan como planos 3D translúcidos con patrón de hatch. Se integran en `Scene3D.tsx`.
